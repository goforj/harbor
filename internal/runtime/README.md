# App Runtime

`internal/runtime` provides ordered startup/shutdown hooks and app-level timeout policy for application resources.

## Phases

- `BeforeStartup`
- `Startup`
- `AfterStartup`
- `BeforeShutdown`
- `Shutdown`
- `AfterShutdown`

Startup phases run in registration order.  
Shutdown phases run in reverse registration order.

## How To Register Hooks

Primary extension point: `app/lifecycle.go`.

That file is render-once and built by Wire, so you can inject repositories/services into `NewLifecycleRegistry(...)` and use them in `Register(...)`.

Hooks are registered on the lifecycle manager from `app/lifecycle.go`:

```go
func (r *LifecycleRegistry) Register(lifecycle *runtime.Lifecycle) {
	lifecycle.On(runtime.BeforeStartup, r.BeforeStartup)
	lifecycle.On(runtime.Startup, r.Startup)
	lifecycle.On(runtime.AfterStartup, r.AfterStartup)
	lifecycle.On(runtime.BeforeShutdown, r.BeforeShutdown)
	lifecycle.On(runtime.Shutdown, r.Shutdown)
	lifecycle.On(runtime.AfterShutdown, r.AfterShutdown)
}
```

## Where To Register

Use `app/lifecycle.go`:

Example:

```go
type LifecycleRegistry struct {
	userRepo *users.UserRepo
}

func NewLifecycleRegistry(userRepo *users.UserRepo) *LifecycleRegistry {
	return &LifecycleRegistry{userRepo: userRepo}
}

func (r *LifecycleRegistry) Startup(ctx context.Context) error {
	return r.userRepo.SeedDefaults(ctx)
}

func (r *LifecycleRegistry) Shutdown(ctx context.Context) error {
	return nil
}
```

## Notes

- Keep hooks idempotent.
- Keep shutdown hooks fast and context-aware.
- Return errors when startup cannot continue.
- Shutdown errors are aggregated.

## Apps and Runtime Ports

`apps.go` is generated by GoForj on render. Do not edit it by hand.

It compiles the app list into the binary so production runtime defaults do not depend on `.goforj.yml` or source directory scanning.

App indexes are deterministic:

- `app` is always index `0`.
- named apps are sorted alphabetically starting at index `1`.

Default ports are derived from the app index:

- HTTP/API port: `3000 + appIndex`
- runtime port block: `10000 + (appIndex * 10)`
- HTTP/app metrics port: `runtimeBase + 0`
- scheduler metrics port: `runtimeBase + 1`

Example generated apps:

| App | Index | HTTP | Metrics | Scheduler metrics |
| --- | ---: | ---: | ---: | ---: |
| `app` | `0` | `3000` | `10000` | `10001` |
| `marketplace` | `1` | `3001` | `10010` | `10011` |
| `backstage` | `2` | `3002` | `10020` | `10021` |
Bundled local development tools avoid the app HTTP range. For example, Grafana defaults to `13001`, leaving `3001` available for the first named app.

Use the runtime helpers when framework or app code needs the current app's defaults:

```go
runtime.HTTPPort()
runtime.MetricsPort()
runtime.SchedulerMetricsPort()
```

Use `ForApp` helpers only when tooling needs to inspect another app:

```go
runtime.HTTPPortForApp("marketplace")
runtime.MetricsPortForApp("backstage")
```

Environment variables remain the override path:

- default app: `PORT`, `API_HTTP_PORT`, `METRICS_PORT`, `API_METRICS_PORT`, `METRICS_API_PORT`, `SCHEDULER_METRICS_PORT`, and `METRICS_SCHEDULER_PORT`
- named apps: `<APP>_PORT`, `<APP>_API_HTTP_PORT`, `<APP>_METRICS_PORT`, `<APP>_API_METRICS_PORT`, `<APP>_METRICS_API_PORT`, `<APP>_SCHEDULER_METRICS_PORT`, and `<APP>_METRICS_SCHEDULER_PORT`


Named app prefixes use uppercase snake case. For example, `marketplace` uses `MARKETPLACE_PORT`.

Named app metrics env vars are not generated into `.env`. The compiled app manifest already provides non-conflicting deterministic ports, so add metrics env vars only when you need to override one app.

Named apps intentionally do not consume default-app globals such as `API_HTTP_PORT=3000`; otherwise existing single-app `.env` files would make every app try to bind the same listeners.



## Lighthouse Agent Identity

Lighthouse uses runtime app metadata to identify each connected process without relying on `.goforj.yml`.

Each agent reports:

- `app`: the App, such as `app`, `billing`, or `customer-portal`
- `source`: the logical runtime surface
- `group_key`: the logical runtime group
- `instance_key`: the concrete process instance

The default app keeps compact group keys named after each source. Named apps use `<app>/<source>`, such as `billing/http`.

Instance keys are resolved in this order:

1. `APP_INSTANCE_ID`
2. hostname
3. generated agent ID

For local development, hostname is usually enough. In Kubernetes or any replicated runtime, set `APP_INSTANCE_ID` to a stable pod/process identity when you want Lighthouse to distinguish replicas clearly, for example `worker-01`, `worker-02`, or the pod name.
